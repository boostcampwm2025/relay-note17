## 🎯 Q1. AI를 이용하여 학습 범위 및 주제 그리고 체크리스트를 제안받기

`J057_김용준`

**기존 미션:** 미션 과제를 보고 체크리스트 항목을 좀 보완하기 위해 조언 듣기

**변경 미션:** AI를 이용하여 학습 범위 및 주제 그리고 체크리스트를 제안받기.

### 배경 및 이유

기존 미션은 먼저 체크리스트 초안을 생성 후 피드백을 받는 식으로 진행했다. 그러나 체크리스트 초안 작성 시 과제를 해결하기 위해 어떤 내용을 학습해야 하는지 학습 범위와 학습 수준을 확실히 알 수 없다.

또한 구현 및 설계에 관한 체크리스트는 그것을 세우는 과정 또한 설계의 한 과정이라고 생각한다.

따라서 학습 범위 및 주제 등은 AI를 이용하여 선별하고 구현 및 설계에 관한 체크리스트는 직접 작성하는 것이 바람직하다고 생각했다.

### 목적

짧은 시간 안에 과제에 꼭 필요한 지식을 얻기 위해 적절한 학습 범위 설정하기


### 달성 기준

- 학습 관련 체크포인트를 작성할 때 AI에 미션 과제 및 요구사항을 전달하고 미션 해결에 필요한 학습 주제 제안 받기
- 응답을 참고하여 학습 관련 체크포인트를 작성하기
- (선택) 관심있는 주제가 있다면 해당 주제에 대해서 추가적으로 공부할 만한 개념들을 요청하기

### 프롬프트 작성

프롬프트 작성은 미션 주제 및 요구사항, 학습 목표 등을 참고하여 작성한다. 해당 내용을 복사해서 붙여넣을 수도 있지만 보다 정확한 답변과 분석 능력 향상을 위해 주제와 요구사항을 어느 정도 정리한 후 작성하는 것이 권장된다. 프롬프트 형식은 위 내용을 포함한다면 자유롭게 작성해도 된다.

## 🎯 Q2. 나만의 리드미 초안 작성해주는 AI 프롬프팅하기

`J219_이하은`

### 미션
- 기존 미션: 리드미를 AI 에게 넘겨주고 예상질문 추천 받아 스스로 답해보기
- 개선 미션: 나만의 리드미 초안 작성해주는 AI 프롬프팅하기

### 배경
기존 1주차 미션은 피어 피드백 시간을 대비하는 것을 목적으로 README.md 작성 내용을 바탕으로 발표 준비를 하는 퀘스트였다.

그러나 문제가 너무 복잡하거나, 학습과 구현 시간이 너무 오래 걸리는 경우 README 파일에 어떤 내용을 적어야 좋을지, 어떻게 정리해야 할지 모르고 정리하는 데에도 너무 오래 걸려서 리드미를 제한 시간 내에 못 적는 경우가 생길 수 있다.

실제로 챌린지 과정을 진행하면서, 나를 포함하여 팀원들 중 리드미 파일을 적지 못한 경우를 몇 번 본 적이 있다.

이에 "학습 보조 도구로써 AI 활용"이라는 주제를 바탕으로, 문제 요구 사항과 개념을 정리하면 리드미 파일의 초안을 작성해주는 나만의 리드미 초안 작성 AI 프롬프팅 미션을 만들게 되었다.

### 목적
- AI 프롬프팅 훈련
- 학습 보조 도구로써 AI 사용해보기

### 달성 기준
- 프롬프팅 과정을 간략히 정리한다.
- AI가 작성해준 리드미 초안을 올리고, AI가 작성해 준 초안을 스스로 피드백한다.
- AI가 작성해 준 리드미 초안을 바탕으로 실제 리드미 파일을 완성하고, AI를 사용하지 않을 때와 비교해 어떤 장단점이 있었는지 정리한다.

### 선택 사항
- 유료 GPT 이용자라면, 나만의 리드미 초안 작성 GPTs를 만든 후 Slack에 공유해 본다.

    <br>→ 학습 범위를 명확히 정하고 시작하면 시간을 절약하고 구현 효율도 높일 수 있음

    <br>→ AI의 시각을 빌려 체크포인트 인사이트를 얻으면, 놓친 부분을 발견하거나 새로운 관점을 확보하는 데 도움이 될 것이라 기대함

  3. 최종 체크포인트를 기준으로, AI에게 학습할 개념/주제 및 학습 범위 추천받기
     - AI가 추천한 범위를 바탕으로 학습 진행
     - 구현 이후 여유가 있으면, 추가 학습이 필요한 주제도 AI에게 받아서 추가 학습 진행

#### `J299_황재호`

- **수행할 퀘스트** : Q4. 문장 바로잡기

- **선정 이유**

그동안 학습한 내용을 옳바르게 이해했는지에 대한 검증 작업이 필요하다고 생각했다. 이에 AI를 활용해서 틀린 정보를 학습했는지 확인해보고자 한다.

- **첫번째 질문**

  > Git에서 commit 객체는 변경된 파일의 blob 해시만 저장하며, 디렉토리 구조는 저장하지 않는다.

- **답변**

  > commit 객체는 변경되지 않은 blob 해시 또한 index를 기준으로 모두 저장하며, index로 루트 디렉토리부터 tree 객체와 blob 객체를 생성하여 해당 git 디렉토리의 구조를 모두 저장한다.

- **두번째 질문**

  > Git에서 add 명령어는 파일의 내용을 commit 객체에 바로 저장하며, 이후 commit 명령어는 log에 해당 파일의 경로와 변경 내용을 기록한다.

- **답변**

  > add 명령어는 파일의 내용을 commit 객체가 아닌 index에 먼저 저장한다. 이를 스테이지에 파일을 올린다고 표현하며, 이후 commit 명령어를 통해 `.git/objects/` 폴더에 새로운 blob, tree, commit 객체를 저장하고 log에 커밋 이력을 기록한다.

- **세번째 질문**

  > git add 명령어는 새로운 커밋 객체를 생성하고, 해당 파일의 blob 객체를 .git/objects/에 바로 저장한다. 이때 기존 커밋은 삭제되고 새로운 트리 구조가 생성되어 로그에 반영된다.

- **답변**

  > git add 명령어가 아닌 git commit 명령어 실행 시 새로운 커밋 객체를 생성한다. 기존 커밋은 삭제되지 않고 계속해서 모든 객체들이 `.git/objects/`에 저장되며 새로운 트리 구조를 생성하고 루트 트리 객체를 생성한 커밋 객체에 연결한다.

- **느낀 점** :
  다시 한번 각 객체들의 정의와 명령어들을 실행했을 때의 흐름을 확인할 수 있었다. 역시 학습한 내용을 검증하는 방법은 남에게 설명할 수 있는 수준이 되어야 한다라는 기준으로 검증하는 것이 좋다라는 점을 다시 한번 상기시킬 수 있었다.
